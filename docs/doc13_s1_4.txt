Section 1 — Phase 1 Module Scoring Algorithms The Phase 1 deep assessment produces two raw scores — one for the TMJ module and one for the cervical module — which are then normalised as percentages of their respective module maximums. These normalised scores drive the profile type classification in Section 2. All scoring logic runs server-side. All threshold values are named constants imported from /content/scoring-thresholds.ts — never hardcoded inline.  ⚠ Schema Correction — Document 7 Update Required The cerv_floor_relief_test column is defined as BOOLEAN in Document 7. This must be changed to VARCHAR(10) to support graduated scoring (3 points for clear reduction, 1 point for slight reduction). The column stores one of three string values:    &apos;clear&apos;  — clear tinnitus reduction after 10 minutes supine  →  3 points   &apos;slight&apos; — slight tinnitus reduction after 10 minutes supine  →  1 point   &apos;none&apos;   — no reduction                                       →  0 points  NULL is treated as none (0 points). Update the Supabase column type to VARCHAR(10) on the first day of the build before any Phase 1 scoring logic is written.  1.1 — When Scoring Runs Scoring runs at two points: **Progressively** — tmj_raw_score and cerv_raw_score are recalculated and written to phase1_assessment on every module save. This keeps stored scores current as the member works through modules. **Finally** — when the member completes the asymmetry module and confirms their profile, both raw scores and both normalised scores are calculated one final time. profile_type, tmj_protocol_assigned, cerv_protocol_assigned, and profile_paragraph are all written at this point. completed_at is set. After this, scores do not change. The profile type classification (Section 2) and all edge case logic (Section 3) run only at final scoring — not progressively.  1.2 — Overlapping Indicator Rule Four indicators exist in both the intake test import and the Phase 1 in-platform history question. These are the same underlying symptom captured at two different stages. They must not double-count. The rule is:  Overlapping Indicator Rule For each of the four overlapping indicators: score from the Phase 1 assessment column if it has been answered (NOT NULL). If the Phase 1 column is NULL — meaning the member has not yet reached that question — fall back to the intake import value.  The Phase 1 answer is always the more carefully considered response and supersedes the intake import for scoring purposes once entered. Section 2 — Profile Type Classification &amp; Protocol Assignment Profile type is determined by comparing the two normalised scores after both modules are complete. The decision tree evaluates conditions in strict top-to-bottom order — the first condition that matches determines the profile type. This ordering is critical: without it, a member with TMJ 65% and cervical 35% could match multiple conditions simultaneously.  2.1 — The Seven Profile Types profile_type Value Trigger Condition Plain English TMJ_DOMINANT tmjNorm &gt; 60 AND cervNorm &lt; 20 Clear jaw driver, negligible cervical involvement CERV_DOMINANT cervNorm &gt; 60 AND tmjNorm &lt; 20 Clear cervical driver, negligible jaw involvement DUAL_DRIVER Both &gt; 30 AND difference &lt;= 15 Both pathways meaningfully and equally active TMJ_PRIMARY_STRONG_SECONDARY tmjNorm &gt; 50, cervNorm 30–50 Jaw leads clearly, substantial cervical involvement CERV_PRIMARY_STRONG_SECONDARY cervNorm &gt; 50, tmjNorm 30–50 Cervical leads clearly, substantial jaw involvement TMJ_PRIMARY_WITH_SECONDARY tmjNorm leads, cervNorm 20–30 Jaw leads, mild cervical involvement present CERV_PRIMARY_WITH_SECONDARY cervNorm leads, tmjNorm 20–30 Cervical leads, mild jaw involvement present  Why 7 types, not 5 Document 7 listed 5 profile type values. Document 13 uses 7. The dominant driver is encoded directly in the profile_type string (e.g. TMJ_PRIMARY_STRONG_SECONDARY vs CERV_PRIMARY_STRONG_SECONDARY). This means session construction, profile paragraph selection, and protocol option recommendations can all switch on profile_type alone without a secondary score comparison. Every place in the codebase that needs to know which side is dominant reads profile_type directly — no additional logic required.  2.2 — Classification Decision Tree Evaluated top-to-bottom. First match wins. All threshold values are named constants from /content/scoring-thresholds.ts. Section 3 — Edge Cases &amp; Asymmetry Pattern Logic Four edge case checks run at profile generation, after both module scores are calculated and before the profile type is finalised. They run in a defined order. Each check either modifies the profile output text, adds a flag to the profile, or (in the case of low confidence) modifies the protocol assignment itself.  3.1 — Edge Case 1: Low Confidence — Both Scores Below Threshold This is the most important edge case. A member with both normalised scores below 20% has not produced confirmatory physical test findings in either module. The assessment did not generate a clear somatic picture. FUNCTION classifyAsymmetryPattern(assessment):    findings = []    IF assessment.asym_jaw_drift_direction IS NOT NULL     findings.push({ structure: &apos;jaw_drift&apos;, side: asym_jaw_drift_direction })    IF assessment.asym_masseter_dominant_side IS NOT NULL     findings.push({ structure: &apos;masseter&apos;, side: asym_masseter_dominant_side })    IF assessment.asym_shoulder_elevated_side IS NOT NULL     findings.push({ structure: &apos;shoulder&apos;, side: asym_shoulder_elevated_side })    IF assessment.asym_scm_dominant_side IS NOT NULL     findings.push({ structure: &apos;scm&apos;, side: asym_scm_dominant_side })    IF findings.length = 0     RETURN &apos;NO_ASYMMETRY&apos;    tinnitusUnilateral =     assessment.asym_tinnitus_worse_ear IS NOT NULL     AND assessment.asym_tinnitus_worse_ear != &apos;bilateral&apos;    sides = findings.map(f =&gt; f.side)   allSameSide = sides.every(s =&gt; s = sides[0])    IF allSameSide AND findings.length &gt;= 2     IF tinnitusUnilateral       RETURN &apos;UNILATERAL_COHERENT&apos;     ELSE       RETURN &apos;STRUCTURAL_ASYMMETRY&apos;    IF NOT allSameSide AND findings.length &gt;= 2     RETURN &apos;MIXED_ASYMMETRY&apos;    RETURN &apos;SINGLE_FINDING&apos;  // Only one asymmetric finding  // Note: CONTRALATERAL_PATTERN is not returned here — it is applied as an // additive flag on top of whichever pattern above applies.  END FUNCTION  Pattern Value Meaning Protocol Effect UNILATERAL_COHERENT Multiple asymmetric findings all point the same direction + unilateral tinnitus Asymmetric protocol emphasis toward dominant side throughout Phase 3 STRUCTURAL_ASYMMETRY Multiple asymmetric findings all same direction but bilateral/central tinnitus Side-specific emphasis applied, lower confidence in lateralisation MIXED_ASYMMETRY Findings point in different directions Protocol addresses each finding individually — no single-side emphasis SINGLE_FINDING Only one asymmetric finding Noted in profile, mild side emphasis where relevant NO_ASYMMETRY No significant asymmetric findings Protocol bilateral throughout — both sides worked equally  3.5 — Edge Case Check Order at Profile Generation FUNCTION runAllEdgeCaseChecks(assessment, user, tmjNorm, cervNorm):    flags = {}    // 1.
